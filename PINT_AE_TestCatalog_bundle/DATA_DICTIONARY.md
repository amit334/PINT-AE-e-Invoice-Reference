# Data Dictionary — `PINT_AE_TestCatalog.csv`

> The authoritative column reference for the UAE PINT-AE scenario catalog (`third_round`).
> Supersedes the older column list in `.claude/rules/scenario-axes.md §7`. A newcomer should
> be able to consume the CSV from this file alone. Cross-refs: [`COMPLETENESS_CERTIFICATE.md`](./COMPLETENESS_CERTIFICATE.md)
> (what is proven), [`ID_SCHEME.md`](./ID_SCHEME.md) (the `TestCaseID` contract),
> [`DEFECT_REGISTER.md`](./DEFECT_REGISTER.md) (schema defects + the 8 documented rows).

**File:** `PINT_AE_TestCatalog.csv` (UTF-8, comma-delimited, RFC-4180 quoting) and its identical
`.xlsx` twin. **25 columns · 746 rows.** Positive rows first, then negatives, then domain-notes.

**Do NOT hand-edit the CSV/xlsx** — they are generated by `assemble.py` → `build_bundle.py` from
the spec sources. Edits go in the specs; the catalog is rebuilt.

---

## Row population at a glance
The catalog holds three **`Kind`s**, and many columns are populated only for some kinds:

| Kind | Rows | Engine outcome | `ExpectedRuleID` | `RejectSource` |
|------|------|----------------|------------------|----------------|
| **Positive** | 351 | PASS | blank | blank |
| **Negative** | 358 | FAIL (or DOCUMENTED) | set (the target rule) | `schematron` |
| **Domain-note** | 37 | PASS (engine can't catch) | blank | `domain-business-rule` (26 SV/DI) · `domain/policy` (11 Class-B) |

"Derived" below = computed from the instance XML or another column by the build, not authored by
hand. "Machine-verified" = the value is checked by `verify_instances.py` and/or
`validate_against_schematron.py`, so it cannot silently drift.

## Columns (in file order)

### 1. `TestCaseID` — *authored, stable, unique key*
The primary key. Unique across all 746 rows and **stable** — downstream exercises (e.g. the
entry-channel testing) reference catalog rows by this key. Prefix taxonomy (full contract in
[`ID_SCHEME.md`](./ID_SCHEME.md)): `EX:` native vendor example · `NEG:` per-assert negative ·
`PX-…` patched positive (incl. cycle series `PX-C3/C4/C5/C8`) · `CV:` carrier-variation negative ·
`SVDI…` schema-valid/domain-invalid note · `CLASSB…` behavioural policy note · plus the
arithmetic/cardinality families `CARD-*`, `DEC-*`, `LINE-*`.

### 2. `Kind` — *authored*
Enum: **`Positive`** · **`Negative`** · **`Domain-note`**. The top-level row class (table above).

### 3. `Subtype` — *authored*
Provenance / generation method. Enum (live counts): `patched` (315), `per-assert` (294),
`carrier-variation` (56), `vendor-example` (32), `svdi` (26), `class-b` (11), `documented` (8),
`custom-seed` (4). `per-assert` + `documented` = the 302 bijection negatives; `svdi` + `class-b` =
the 37 domain-notes.

### 4. `TestGroup` — *authored*
Functional bucket for filtering/reporting (14 values), e.g. `Party/Identifier`, `Tax-Category`,
`Transaction-Flag`, `Currency/FX`, `Allowance-Charge`, `References/Dates`, `Payment`,
`Codelist-Value`, `Arithmetic/Totals`, `Item/Classification`, `Cardinality`, `Doc-Type`,
`Boundary/Format`, `Domain/Policy`. Not a validity axis — an organisational lens.

### 5. `DocTypeCode` — *authored (key axis)*
The UBL `InvoiceTypeCode` / `CreditNoteTypeCode`. Enum: **`380`** Tax Invoice · **`381`** Tax Credit
Note · **`480`** Commercial Invoice · **`81`** Commercial Credit Note · **`389`** Self-Billing Invoice ·
**`261`** Self-Billing Credit Note · **`n/a`** (the 8 documented rows that build no instance).
⚠️ The UNCL1001 codelist *names* contradict the UAE business labels — **key off the code**
(see `.claude/rules/schema-document-type-codes.md`). This is the single axis that drives most others.

### 6. `DocClass` — *derived from `DocTypeCode`*
Human-readable doc class — a lookup, **not** an independent axis: 380→Tax Invoice, 381→Tax Credit
Note, 480→Commercial Invoice, 81→Commercial Credit Note, 389→Self-Billing Invoice, 261→Self-Billing
Credit Note, `n/a` for the documented rows.

### 7. `TaxCategory` — *authored / derived; machine-verified*
The line/breakdown VAT category. Single values: `S` Standard · `Z` Zero-rated · `E` Exempt ·
`O` Outside-scope · `AE` Reverse-charge · `N` Margin. Multi-line positives carry combined values
(`S+Z`, `S+E`, `S+AE`, `E+O+Z`, `S+E+O+AE+Z`). Deliberate fault values appear on a few codelist
negatives (`ZZ`, …). `n/a` on the documented rows. ⚠️ Margin category `N` is encoded as Latin "N"
(U+004E) in instances — the schematron uses Latin N; the `.gc` codelist's Greek-Nu is a known
divergence (see `DEFECT_REGISTER.md`). When an instance is built, this value is read back from the
XML so it cannot drift from the row.

### 8. `TransactionFlags` — *authored / derived; machine-verified*
The 8-bit transaction-type string, or the literal `Standard` (= `00000000`). Bit order:
pos1 FreeTradeZone · 2 Deemed · 3 Margin · 4 Summary · 5 Continuous · 6 DisclosedAgent ·
7 e-Commerce · 8 Exports. Derived back from the instance on verification.

### 9. `ItemType` — *authored; machine-verified*
`G` Goods · `S` Services · `B` Both · `n/a` (when the row does not exercise item classification or
builds no instance). Drives HS-code / SAC field expectations.

### 10. `Currency` — *authored; machine-verified*
ISO currency of the document. `AED` (majority), foreign samples `USD`/`EUR`/`GBP`, deliberate
codelist-fault values (`XXX`, `ZZZ`) on currency negatives, `n/a` on documented rows. Foreign
currency triggers the AED tax-accounting / exchange-rate conditional fields.

### 11. `Scenario/CondFields` — *authored*
Free-text description of the scenario and any conditional fields it must carry (e.g. "Summary ⇒
InvoicePeriod", "Exports ⇒ Deliver-to non-AE"). For negatives, states the fault being injected.

### 12. `SeedXML` — *authored*
The vendor seed instance the row is patched from, relative to `UAE_E-Invoicing_Docs/Schemas/_index/examples/`
(e.g. `pint-invoice/Standard tax invoice.xml`). Blank for the documented rows (no instance built).

### 13. `InstanceXML` — *derived (build output)*
Path to the built instance under `third_round/catalog_xml/` (e.g. `catalog_xml/NEG_ibr-068.xml`).
This is the artifact the schematron actually validated. Blank for the 8 documented rows.

### 14. `Mutation/Build` — *authored / build recipe*
The exact, replayable patch applied to `SeedXML` to produce `InstanceXML` (element path + operation,
e.g. `set /Invoice/cbc:CustomizationID = …`). For native vendor examples this is blank (the seed *is*
the instance). This column makes every instance reproducible by inspection.

### 15. `PredictedOutcome` — *authored (the oracle prediction)*
What the catalog asserts the engine will do, **before** running it. Forms: `VALID` (positives);
`INVALID (fires <rule>)` (executable negatives); `INVALID` (the 8 documented negatives — engine
never reaches the rule); `INVALID(domain)-engine PASSES` (the 37 domain-notes — the engine PASSES,
domain rejects). Reconciled against `ActualOutcome` by the validator.

### 16. `ExpectedRuleID` — *authored; machine-verified for negatives*
The single schematron assert id this negative is built to trigger (e.g. `ibr-068`, `ibr-152-ae`,
`aligned-ibrp-s-08`). **Set for all 358 negatives** (302 distinct ids = the bijection; the 56
carrier-variations reuse covered ids). **Blank for positives and domain-notes.** The validator
confirms this id appears in `FiredRuleIDs` for verified negatives.

### 17. `RuleLayer` — *authored / derived*
Which schematron file the `ExpectedRuleID` lives in: **`BASE`** (170, PINT-UBL preprocessed) or
**`AE`** (132, jurisdiction-aligned). 170+132 = the 302 bijection asserts. **Blank** for positives,
domain-notes, and the 56 carrier-variations (which reuse a covered id).

### 18. `BusinessTerms` — *authored / reference*
Comma-separated PINT business-term ids (e.g. `IBT-029,IBT-030,…`) the rule/scenario touches —
traceability back to the PINT semantic model. Populated mainly on per-assert negatives.

### 19. `RuleMessage` — *reference (from the `.sch`)*
The human-readable assertion message text quoted from the schematron, for the negative's target
rule. Helps a reviewer understand *why* the rule fires without opening the `.sch`.

### 20. `RejectSource` — *authored; machine-consistent*
Who rejects the row. **`schematron`** (350 verified negatives — the engine catches it) ·
**`domain-business-rule`** (26 SV/DI notes — schema permits, UAE policy forbids) ·
**`domain/policy`** (11 Class-B behavioural notes) · **blank** (351 positives + the 8 documented
rows). For the 8 documented rows the rejection is intended-but-engine-unreachable (see `Notes`).

### 21. `ActualOutcome` — *derived (machine-verified); the measured result*
What the compiled schematron **actually** did when the instance ran: **`PASS`** (388 = 351 positives
+ 37 domain-notes) · **`FAIL`** (350 verified negatives) · **blank** (8 documented rows — never run).
Produced by `validate_against_schematron.py`. A `PASS`/`FAIL` that disagreed with `PredictedOutcome`
would be a reconciliation finding; there are **0**.

### 22. `FiredRuleIDs` — *derived (machine-verified)*
The full set of fatal asserts that fired on the instance (semicolon/comma-separated), parsed from the
SVRL. For verified negatives it contains `ExpectedRuleID` (+ any documented-expected cascades — see
`cofire_triage.tsv`). Empty for PASS rows. This is the raw evidence behind the single-fault accounting
(205 single-fault / 145 documented-cascade).

### 23. `Status` — *derived; the verification gate*
**`VERIFIED`** (738) — built, run, and reconciled green. **`DOCUMENTED`** (8) — a real fatal assert
that is genuinely unreachable through the harness (dead stub / compiler-dead / XSLT-runtime-crash /
predicate-shadowed); kept for bijection completeness with a typed reason in `Notes`, **NOT
machine-verified by construction**. Only the 738 `VERIFIED` rows are in the 738/738 schematron run.

### 24. `SourceRef` — *authored; 100% populated*
Provenance citation for the row — a schematron layer (`schematron BASE` / `schematron AE`), a domain
source (`Guidelines §10.3`, `MD243 Art9`), or `example set` for native vendor examples. **No row is
blank** (audited), giving full traceability from every catalog row to an authority.

### 25. `Notes` — *authored*
Free-text annotations: documented-expected co-fire explanations ("co-fires EXPECTED (…): reason"),
the typed unreachability reason for each of the 8 documented rows
(`DEAD` / `DEAD-IN-COMPILED-XSLT` / `XSLT-RUNTIME-CRASH` / `PREDICATE-SHADOWED`), and SV/DI rationale.

## Quick consumption recipes
- **Only the machine-verified set:** filter `Status = VERIFIED` (738 rows).
- **Positives / negatives / notes:** filter `Kind`.
- **The 302-assert bijection:** `Subtype ∈ {per-assert, documented}`; the rule is `ExpectedRuleID`.
- **Engine-PASS-but-domain-invalid:** `Kind = Domain-note` (37) — these need a non-schematron control.
- **Reproduce a single instance:** apply `Mutation/Build` to `SeedXML` → compare to `InstanceXML`.
